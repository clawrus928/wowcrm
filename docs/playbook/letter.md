# 給未來 session 的信

寫於 2026-07-04,Fable 5 session。這是唯一一次由這個檔次的模型立制度;之後由你(Sonnet / Opus / Haiku)長期執行。這封信講檔案裡放不下的三件事、這套制度會怎麼壞、以及我對自己產出的誠實評估。

## 一、使用者沒問,但我認為最重要的三件事

**1. 這個系統的價值槓桿在「驗證紀律」,不在模型檔次。**
本 repo 的錯誤史(見 lessons.md)沒有一條是「模型不夠聰明」造成的,全部是「沒驗到」:build 不覆蓋後端、刪除沒級聯、幣別混加。Sonnet 照著 judgment.md 第 2 節的完成定義走,產出品質不會輸給不驗證的 Fable。反過來,任何檔次的模型跳過驗證,都會把錯誤直接部署到澳門業務團隊正在用的生產系統——這裡 PR 合併即部署,沒有 staging。

**2. 有一個未收尾的工作在等你。**
`claude/wowcrm-upgrade-qq4bn` 分支上有 4 個已驗證的 audit commits(幣別修正、刪除警告、待辦跟進卡、文件編號等,commit ec76f85..b127a18)+ 本 playbook,**尚未開 PR**。使用者的習慣是:先開 PR → 使用者說「merge」→ 合併觸發部署。接手的第一件事是問使用者要不要開這個 PR,不要擅自合併。

**3. 額度問題誠實交代。**
使用者問過「被安全機制導向 Opus 4.8 的請求是否消耗本窗口額度」——**未確認,我查不到,建議到 usage 儀表板實測**。同樣未確認的:Agent 工具的 `model: "fable"` 在未來 session 是否可用。所以 model-dispatch.md 把 opus 當最高檔規劃,fable 出現時是 bonus 不是依賴。

## 二、這套制度最可能的退化方式與預防

1. **lessons.md 變垃圾場**:什麼都記、沒人蒸餾,最後沒人讀。預防:maintenance.md 第 2 節限定三種寫入時機、第 3 節 150 行強制蒸餾。你要真的執行,不是知道有這條。
2. **範本被儀式化**:抄 templates.md 但驗收條件留空或寫「功能正常」這種不可判定的話。預防:templates.md 尾註那句是硬規則——寫不出可機械判定的驗收 = 還沒想清楚,不准交辦。
3. **規則被「本次例外」蠶食**:每次都有理由跳過冒煙測試、跳過 fresh-context 審查,三次之後制度名存實亡。預防:例外可以有,但每次例外必須在 lessons.md 記一行「何時、為何跳過、結果」;不肯記就不准跳。
4. **CLAUDE.md 重新長胖**:大家把新規則往常載檔塞。預防:超 150 行即抽離(maintenance.md 第 3 節);新內容預設進 playbook 按需檔,CLAUDE.md 只加一行路由。

## 三、我對自己產出的信心排序(誠實條款)

- **信心最低:diagnosis.md 的「前三名」排名。** 它來自一個 session 的體感,不是量測。方向我有把握(整檔重讀、原始輸出、主模型下場確實最浪費),但順序與量級是推估。用兩三個 session 後若體感不符,照 maintenance.md 流程改它。
- **次低:model-dispatch.md 第 5 節的升降級門檻。**「haiku 錯 1 次升級、sonnet 連錯 2 次升級、最多重試兩輪」是我拍的參數,原則(快升、帶失敗軌跡、有上限)比數字可靠。數字可調,原則不要丟。
- **中等:templates.md。** 結構我有把握,但真正的驗收條件品質取決於填空的人。範本防不了敷衍。
- **信心最高:judgment.md 第 2 節(完成定義)與第 3 節(何時問使用者)、lessons.md 的四條教訓。** 這些直接來自本 session 實際踩過、驗證過的事實。
