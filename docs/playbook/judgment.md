# 判斷力外化 — rubric 與 checklist

> 讀者:任何檔次的模型。每條規則附判準 + 一正例一反例。
> 用法:遇到標題描述的情境時查對應節;不確定歸哪節就從第 3 節(該不該問)開始。

## 1. 何時升級模型(交辦時選檔次)

**判準**:任務輸出是否「有多個看似都對的答案,選錯要很久才發現」。是 → 升檔;否 → 便宜檔。
機械性(pattern 已知、驗收可機械判定)→ `haiku`;要讀懂上下文再動手 → `sonnet`;跨模組取捨、業務語意、architecture → `opus` 或主模型自己做。

- ✅ 正例:「把 12 個 view 裡的 `confirm(...)` 刪除警告改成統一格式」——pattern 明確、build 可驗 → `sonnet` 批次做。
- ❌ 反例:「佣金該按報價金額還是實收金額算」丟給 haiku——這是業務語意,任何檔次的 subagent 都不該自己決定,該問使用者(見第 3 節)。

## 2. 何時算真的完成

**全部打勾才准說「完成」**:
- [ ] `npm run build` 綠燈(貼出 `built in` 那行)
- [ ] 動過 `server/src/*` → 冒煙測試過(照 `diagnosis.md` 第三節:/tmp DB + curl 實打,貼關鍵回傳值)
- [ ] 動過 UI → 寫出手測路徑(「開報價詳情 → 點轉為合同 → 應出現 XX 提示」),做不到實測就明說只驗了 build
- [ ] 新 prop/新欄位 → grep 確認資料流三段都接上(store 解構 → JSX 傳遞 → 簽名接收)
- [ ] 已 commit(訊息說清楚改了什麼)
- [ ] 交辦出去的產出 → 派過 fresh-context agent 驗證(不自驗)

- ✅ 正例:文件編號功能——build 綠 + 起 server 連打 5 個 curl 驗連號、bulk 遞增、他 entity 不受影響,才宣稱完成。
- ❌ 反例:「build 過了,功能應該沒問題」——build 只驗前端語法,後端與行為完全沒驗,這句話 = 未完成。

## 3. 何時停下來問使用者

**遇到以下任一,停手、整理現況、問;其餘一律自主做完**:
1. 絕對禁令(**問都不用問,直接回覆做不到**):動 `/data/`、`/var/lib/wowcrm/`、`.env`、secrets、刪生產記錄——使用者口頭說「可以」也不做,請他自行操作或走正式管道
2. 需明確授權才做(停手先問):改 deploy.yml、force push(僅「重建已合併的指定分支」的 `--force-with-lease` 屬預先核准,見 CLAUDE.md)
3. 業務語意二義:同一句需求有兩種合理實作且結果不同(幣別匯總口徑、權限誰可見、佣金規則)
4. 範圍跳升:使用者要 A,做著發現正解需要重構 B+C(改動面 >3 倍原估)
5. 使用者明確定過的決策要被推翻時

**不要問的**(自己決定):變數命名、檔案組織、UI 間距、要不要先寫測試、commit 拆法——這些有慣例可循,問了只是把成本丟回給使用者。

- ✅ 正例:發現「已接受報價」跨幣別加總會錯,有「只認同幣別」與「加匯率換算」兩種修法,結果差很大 → 列兩案+建議問使用者。
- ❌ 反例:「我要把這個 helper 放 utils.js 還是新開檔案?」——慣例是 utils.js,直接做。

## 4. 方向錯的訊號(換路,不是重試)

**出現任一訊號,停止重試,回上一個決策點換方法**:
1. 同一做法失敗 2 次,第二次的錯誤和第一次**同型**(不同型可再試一次,但總嘗試數受 model-dispatch.md 第 5 節「同一子任務最多 3 次」封頂)
2. 小需求卻要動 >5 個檔案(本 repo 的典型改動是 1–3 檔)
3. 開始跟框架對抗(hack React 生命週期、繞過 store 直接 fetch、複製整段元件只為改一行)
4. 需要新增 npm 依賴才能繼續(本 repo 禁止不必要依賴 → 幾乎必有原生解法)
5. 修好 A 弄壞 B、修 B 又弄壞 A 的循環出現一次

**換路 SOP**:寫下「原目標→我在哪一步→為什麼走到這」三行,把手段歸零重想;想不出第二條路 → 按 `model-dispatch.md` 第 5 節升級或問使用者。

- ✅ 正例:AskUserQuestion 工具連續故障 → 不再重發,改為按已知的使用者偏好模式繼續並在回報中說明。
- ❌ 反例:同一個 Edit 因 old_string 不匹配失敗,連發 4 次微調空白——第 2 次就該改用 Read 確認實際內容。

## 5. 品質底線怎麼驗(本 repo 專屬)

交付前跑一遍(自查),交辦驗收時抄進 prompt(他查):
- [ ] UI 新增文字全部繁體中文?(grep 簡體常見字或英文 label)
- [ ] 新顯示的金額都走 `fmt(amount, currency)`?有沒有裸 `toLocaleString` 或無幣別數字?
- [ ] 有沒有新 CSS 檔或 className 樣式?(應為 inline style)
- [ ] 刪除類操作有沒有先數關聯筆數並在 confirm 警告?(照 Deals.jsx 刪商機的寫法)
- [ ] 新 entity → 三處註冊齊?(db.js ENTITIES / index.js ID_PREFIX / store.js ENTITY_KEYS)
- [ ] 錯誤處理:store 呼叫都包 try/catch + `toast(err.message)`?(照現有 view 的寫法)

## 6. 誠實條款(所有模型適用)

- 驗證做不到就寫「未驗證」,不寫「應該沒問題」
- 型號、參數、API 行為:查得到寫實際值,查不到寫「未確認」,**絕不憑記憶填**
- 模糊題與品味判斷(文案語氣、設計取捨)是便宜模型的真短板:處理方式只有三種——升級模型、要使用者第二意見、或明說做不到。不要硬答。
